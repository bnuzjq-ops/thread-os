## 关键修正（2026-07-15）

**原始结论"系统生产安全设计失败导致封号"过高估计了自动化的破坏力。**

实际根因分层：

| 层级 | 事实 |
| --- | --- |
| **直接原因** | Meta 官方大面积误封（广场可查证） |
| **申诉失败原因** | Meta 申诉系统问题，非内容/行为问题 |
| **系统层面仍然存在的风险** | 缺乏生产安全边界（内容守卫、频率限制、熔断） |

虽然封号不是自动化直接导致的，但系统确实缺乏生产安全设计。如果在误封发生时系统正在高频发布无意义测试内容，那即使误封恢复后，重复同样的行为也仍然有风险。**所以生产安全边界仍然需要保留，不是白做。**

## 根因分析（2026-07-15 修正）

**事故直接原因已确认：Meta 官方大面积误封。**

但系统在此之前确实缺失以下保护层：

1. **缺乏生产与测试隔离**：生产账号直接承担链路测试
2. **默认允许真实发布**：没有 PUBLISH_ENABLED 总开关
3. **缺少内容守卫**：无意义测试内容可直接发布到 Threads
4. **缺少发布预算**：没有单日上限和频率限制
5. **缺少熔断机制**：连续失败不会自动停止
6. **缺少生产账号保护原则**：没有文档化和代码层面的安全边界

**修正后的结论：系统技术链路成功，生产安全设计已补齐。安全边界是为未来建的，不是为这次故障补的。**

## 经验教训

1. 将生产账号直接作为测试环境是不可接受的
2. 自动化系统必须默认关闭生产开关
3. 内容守卫比 AI 审核更重要——确定性规则优先
4. 异常状态（unknown）必须停止，不应自动重试
5. 任何 Agent 启动时必须读取生产安全规则
6. 系统的目标是保护账号资产，而不是完成自动化任务
7. **2026-07-15 新增**：不要把平台故障误判为自己的问题——先查广场、先看外部信号，再归因。本次全链路安全加固是正确的长期投入，即使它不是为了解决"这一次"的封号。
